# Database Migrations

## Overview

Database schema changes are managed with **Alembic**. Never edit the database schema manually — always create a migration file.

## Setup

Alembic is initialised once at project setup:

```bash
cd backend
alembic init migrations
```

`alembic.ini` and `migrations/` are committed to the repo. The `migrations/versions/` folder is where generated migration files live.

## Workflow

### 1. Change a model in `backend/models/`

```python
# models/ticket.py
class Ticket(Base):
    __tablename__ = "tickets"
    id = Column(Integer, primary_key=True)
    title = Column(String, nullable=False)
    urgency = Column(String, nullable=False)  # ← new field
```

### 2. Generate a migration

```bash
cd backend
source venv/bin/activate
alembic revision --autogenerate -m "add urgency to tickets"
```

This creates a new file in `migrations/versions/`. Always review it before applying.

### 3. Apply the migration

```bash
alembic upgrade head
```

### 4. Roll back if needed

```bash
alembic downgrade -1   # roll back one step
alembic downgrade base # roll back everything
```

## Naming Convention

Migration messages should be short and descriptive:

```
"create users table"
"create tickets table"
"add urgency to tickets"
"add due_date to tickets"
"add role to users"
```

## On Deployment

GitHub Actions runs migrations automatically as part of the backend deploy:

```bash
alembic upgrade head
```

This runs before FastAPI restarts so the schema is always in sync with the code.

## Rules

- Never edit the database schema manually via `psql`
- Always review autogenerated migration files before applying
- Always commit migration files to the repo
- One logical change per migration (don't bundle unrelated changes)

## Environments

Each environment has its own database — migrations must be run separately on each:

| Environment | Database |
|---|---|
| Dev | `ticketing_dev` (runs on your local VM) |
| Test | `ticketing_test` (runs automatically on deploy) |
| Prod | `ticketing_prod` (runs automatically on deploy) |
